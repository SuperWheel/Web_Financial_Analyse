# Web_Financial_Analyse 项目评价与改进建议

> 评估日期：2026-07-18  
> 归档日期：2026-07-18  
> 归档路径：`docs/archive/2026-07-18-项目评价与改进建议.md`  
> 评估范围：v1.0.0 基线及当前工作区中的 post-1.0 在研改动  
> 项目定位：单机本地运行的企业财务报表分析工具  
> 状态：已归档（外部评价 + 仓库核对意见）


## 一、总体结论

Web_Financial_Analyse 已经形成了真实可用的业务闭环，不是仅用于展示技术栈的 CRUD Demo。

当前系统覆盖：

1. 企业档案与三大财务报表录入。
2. PDF、Excel、巨潮在线年报导入。
3. 原始披露行保留、标准科目映射和人工复核入库。
4. 财务比率计算、科目多期对比和趋势展示。
5. Excel 导入、导出及本地 SQLite 数据存储。

按照“个人或小型团队使用的本地财务分析工具”定位，项目综合评价为 **7.5/10**。如果以公开部署、多用户使用的商业 SaaS 标准衡量，目前约为 **5/10**。

项目下一阶段最重要的方向，不是继续快速扩大功能范围，而是将已经具有价值的 A 股 CAS 主路径做得更稳定、更容易回归验证、更容易维护。

## 二、项目的核心优势

### 2.1 产品闭环完整

项目已经贯通以下数据链路：

```text
年报获取
  → PDF / Excel 解析
  → 科目映射与质量提示
  → 人工核对
  → 三表及披露明细入库
  → 比率分析 / 多期对比
  → Excel 导出
```

这意味着项目解决的是完整任务，而不是若干相互独立的功能页面。

### 2.2 财务数据的人审原则正确

导入任务不会自动提交财务数字，而是先进入 review 状态。这一设计符合财务数据对准确性和可追溯性的要求，也为后续展示覆盖率、未映射科目和异常信息提供了合理的交互位置。

### 2.3 L0 与 L1 双层数据设计有长期价值

- L0 披露层保存年报原始科目、金额、页码和映射信息。
- L1 标准科目层支持统一比率计算和跨期比较。

这种设计同时保留了来源证据和标准化分析能力，是项目最有价值的架构决策之一。

### 2.4 后端分层清晰

后端基本遵循：

```text
api → services → models
       ↑
schemas 校验边界，core 保存横切规则
```

年报导入也被拆分为 profile、locate、extract、map、validate 等阶段，便于单独定位识别问题。

### 2.5 工程文档较完整

项目包含 Spec 变更包、架构说明、API 文档、任务清单和持续开发日志。对于个人或小团队项目，这种文档完整度明显高于平均水平，有利于回顾设计理由并减少重复决策。

### 2.6 后端自动化测试基础良好

2026-07-18 本地验证结果：

- 后端共发现 60 个测试点。
- `pytest` 结果为 **60 passed**。
- 测试覆盖企业、三表、比率、对比、Excel、年报导入和在线拉取等主要切片。

## 三、当前主要问题

### 3.1 当前在研前端无法通过编译

当前工作区执行 `vue-tsc` 时，在 `frontend/src/views/ImportView.vue` 第 1126 行产生模板语法错误。

当前表达式：

```vue
{{ row.periods.map((p: { label: string }) => p.label).join('、') }}
```

Vue 模板表达式中不应直接书写该 TypeScript 参数类型。应将格式化逻辑移入脚本函数或移除模板内类型注解。

这意味着当前状态需要区分：

- 后端：60 个测试通过。
- 前端：当前工作区尚不能通过类型检查和完整构建。

### 3.2 前端视图组件过大

当前主要视图的大致规模：

| 文件 | 行数 | 风险 |
|---|---:|---|
| `AnalysisView.vue` | 2,386 | 分析状态、图表和布局高度耦合 |
| `ImportView.vue` | 1,598 | 搜索、下载、队列、核对、提交和 Excel 导入混在一个组件 |
| `CompareView.vue` | 1,286 | 表格、指标、趋势和坐标控制集中 |

这些组件已经超过适合持续迭代的规模。继续直接添加功能，会提高模板损坏、状态竞态和局部修改牵连全页的概率。

### 3.3 缺少自动质量门禁

当前未形成完整的自动质量门禁：

- 未发现持续集成工作流。
- 未形成明确的 ESLint / Prettier 配置。
- 未形成 Ruff / mypy 等后端静态检查。
- 未发现前端单元测试或端到端测试。

现有测试本身有价值，但依赖开发者主动运行，无法在每次提交时自动阻止回归。

### 3.4 数据库迁移机制只适合早期阶段

当前通过启动时检查 ORM 字段，并自动执行 `ALTER TABLE ADD COLUMN` 来补齐 SQLite 列。

该方案可以满足早期新增可空科目字段的需求，但不能可靠处理：

- 字段改名或删除。
- 约束和索引变化。
- 历史数据转换。
- 迁移版本记录与回滚。

随着用户积累真实财务数据，数据库备份和可追踪迁移的重要性会明显上升。

### 3.5 PDF 解析需要黄金样本回归体系

目前已有真实 PDF 测试，但样本文件缺失时测试会跳过。对于项目最核心的解析能力，仅验证“是否能映射出某些字段”仍然不够。

建议为固定样本记录以下黄金指标：

- 公司名称、证券代码、报告年份和金额单位。
- 三表定位结果。
- 核心科目命中率。
- 关键金额及允许误差。
- 未映射科目数量与主要原因。
- 资产负债表平衡检查。
- 解析耗时。

这样才能发现“修复一家公司，却破坏另一家公司”的规则回归。

### 3.6 当前未提交改动规模过大

2026-07-18 检查时，工作区约有 1,970 行新增和 200 行删除，其中 `ImportView.vue` 单文件约增加 990 行。

这种改动规模不利于：

- 快速定位错误来源。
- 独立评审不同功能。
- 出现问题时安全回退。
- 保持提交历史能够解释设计演进。

## 四、建议实施路线

### Phase A：恢复可靠交付状态

目标：任何准备提交的代码都能被统一命令验证。

建议任务：

- [ ] 修复 `ImportView.vue` 当前模板编译错误。
- [ ] 执行后端完整测试并保持 60 个测试通过。
- [ ] 执行前端 type-check 和 production build。
- [ ] 将当前大批量改动拆分为逻辑清晰的提交。
- [ ] 确认 README、架构文档和实际功能状态一致。

验收标准：

```text
backend: pytest 全部通过
frontend: vue-tsc --noEmit 通过
frontend: vite build 通过
git: 每个提交对应一个可描述、可验证的功能或修复
```

### Phase B：建立最小质量门禁

目标：让回归问题在提交或合并前自动暴露。

建议任务：

- [ ] 增加 CI：后端 pytest。
- [ ] 增加 CI：前端 type-check 和 build。
- [ ] 增加基础 ESLint，并先约束新增代码。
- [ ] 增加 Ruff，先启用低争议规则。
- [ ] 为关键前端纯函数补充单元测试。
- [ ] 为“创建企业 → 导入核对 → 入库 → 分析”补充一条端到端烟测。

验收标准：

- 每次提交或合并请求自动执行质量检查。
- 编译错误、后端测试失败和明显静态错误不能进入主分支。
- 质量工具的引入不要求一次性格式化整个仓库。

### Phase C：拆分前端巨型组件

目标：降低页面内部耦合，而不改变用户行为。

#### ImportView 建议拆分

```text
ImportView.vue
├── components/import/CninfoSearchPanel.vue
├── components/import/ReviewQueueNav.vue
├── components/import/MappingQualityPanel.vue
├── components/import/UnmappedLinesTable.vue
├── components/import/StatementReviewEditor.vue
├── components/import/ExcelImportPanel.vue
└── composables/useImportReviewQueue.ts
```

#### AnalysisView 建议拆分

```text
AnalysisView.vue
├── components/analysis/RatioKpiGrid.vue
├── components/analysis/HealthSummary.vue
├── components/analysis/RatioTrendChart.vue
├── components/analysis/AxisControls.vue
└── composables/useRatioAnalysis.ts
```

#### CompareView 建议拆分

```text
CompareView.vue
├── components/compare/CompareKpiGrid.vue
├── components/compare/ChangeRanking.vue
├── components/compare/StatementMatrix.vue
├── components/compare/AccountTrendChart.vue
└── composables/useStatementCompare.ts
```

拆分时应遵循：先提取纯展示组件，再提取状态逻辑；每一步保持类型检查和界面行为不变。

### Phase D：强化数据可靠性

目标：保护已经入库的真实财务数据，并提升解析回归能力。

建议任务：

- [ ] 增加数据库一键备份和恢复功能。
- [ ] 引入正式数据库迁移工具或版本化迁移脚本。
- [ ] 建立固定年报样本集和黄金结果文件。
- [ ] 为每次映射规则修改输出覆盖率差异。
- [ ] 对低覆盖率、报表不平衡、异常金额单位提供明确阻断或二次确认。
- [ ] 允许用户查看某个 L1 数字对应的 L0 原始行、页码和映射规则。

### Phase E：继续产品增强

完成前述稳定性工作后，再按价值排序继续：

1. 导入核对金额可编辑及来源定位。
2. 同比、缺失期间和季度累计值等分析边角处理。
3. 港股 GAAP 独立解析器。
4. EDGAR / US GAAP 独立解析器。

港股和 US GAAP 不应直接复用或污染 CAS 映射规则，而应通过独立 extractor 和词典接入同一 review 管道。

## 五、建议暂缓的事项

现阶段不建议优先投入：

- 多用户和权限系统。
- 云端部署。
- 实时行情。
- 机器学习预测。
- 复杂异步任务平台。
- 大规模 UI 重设计。

这些方向会扩大系统边界，但不能直接解决当前最重要的可靠性和维护性问题。

## 六、建议关注的核心指标

后续迭代可以持续记录以下指标：

| 维度 | 建议指标 |
|---|---|
| 构建质量 | 后端测试通过率、前端构建通过率、CI 通过率 |
| 解析质量 | 三表定位率、核心科目覆盖率、关键金额正确率 |
| 人审效率 | 每份年报需人工修改的字段数、核对耗时 |
| 数据可靠性 | 入库前阻断次数、备份成功率、迁移验证结果 |
| 可维护性 | 巨型组件行数、重复逻辑数量、回归缺陷数 |
| 产品价值 | 从获取年报到完成入库分析的总耗时 |

## 七、最终建议

这个项目真正的竞争力不是比率公式或图表数量，而是以下可信数据链路：

> 公开年报 → 原始披露保留 → 标准科目映射 → 质量提示 → 人工核对 → 可追溯入库。

建议将下一阶段的工程目标定义为：

> 让 CAS 年报主路径达到可重复验证、可解释、可回退和可长期维护的状态。

完成这一阶段后，项目将从“完成度较高的个人项目”提升为“能够长期承载真实财务数据的专业本地工具”。

   - 方向正确且应优先于港股/EDGAR 扩张。

### C. 优先级采纳意见

| 原文 Phase | 是否采纳 | 说明 |

| A 恢复可靠交付 | **立即** | 先修模板 TS；补齐可运行的 type-check/build；拆分超大未提交改动 |

- 引入复杂异步任务平台、多用户、云部署——与 v1.x 非目标冲突。
