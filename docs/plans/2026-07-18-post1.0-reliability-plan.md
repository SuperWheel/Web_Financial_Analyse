## 0. 原则与边界

### 0.1 三原则

1. **先止血再增强**：编译/门禁/提交切片 > 新功能。
2. **小步可回退**：每个切片独立可测、可单独 commit。
3. **不扩系统边界**：不做多用户、云、行情、ML、复杂异步队列（见 `openspec/project.md` 非目标）。

### 0.2 明确不做（本计划周期内）

- 港股 GAAP / EDGAR（顺延至 Phase E，且须独立 extractor）
- 大规模 UI 重设计
- 一次性全仓 Prettier/ESLint 大格式化
- Alembic 全量上线（Phase D 再评估；现阶段先备份 + 脚本化）

### 0.3 当前基线（2026-07-18）

| 项 | 状态 |
|----|------|
| 后端 | `pytest` 收集 **60** 项；主路径测试在 |
| 前端 | `ImportView.vue:1126` 模板内 TS 注解；本机可能缺 `node`，`vue-tsc` 门禁未闭环 |
| 未提交 | ~2000 行级混杂改动（010 批量拉取、公司名识别、核对 UX、文档） |
| P2 部分完成 | 核对页映射质量 / unmapped / 低覆盖确认已做 |
| CI / Lint | 无 |
| 迁移 | 仅 `ensure_sqlite_columns` ADD COLUMN |
| PDF 回归 | 有真实 PDF 测，缺 fixture 则 skip |

### 0.4 统一验收命令（Phase A 起必须能跑通）

```bash
# 后端
cd backend && .venv/bin/python -m pytest -q

# 前端（需可用的 node，或文档约定的 bun 等价方式）
cd frontend && bun run type-check && bun run build
```

建议在仓库根增加 `scripts/check.sh`（Phase A-3），把上述命令收口。

## 4. Phase C — 拆分前端巨型组件（预计 2–4 天，可分周）

**目标**：降耦合；**行为不变**；每拆一步 type-check 绿。

### C0. 拆分纪律

1. 先抽**纯展示**组件（props in / emit out），再抽 composable。  
2. 一次 PR 只拆一个视图或一个子域。  
3. 禁止拆分同时加功能。  
4. 行数目标：容器页 < ~400–600 行；子组件各司其职。

### C1. ImportView（优先，边界已清晰）
